# Pigram Implementation Plan

> **For Hermes:** Lead dev (Opus) orchestrates. Execution delegated to `pi --model kr/claude-sonnet-4.5 --print` subagents, ONE bite-sized task at a time, with spec + code-quality review between tasks. Follow `tdd` (vertical slices), `domain-modeling`, `codebase-design`, and `improve-codebase-architecture` skills throughout.

**Goal:** Rewrite `badlogic/pi-telegram` (fork: `jetmiky/pi-telegram`) into **Pigram** — a clean-architecture, npm-published (`@jetmiky/pigram`), session-local Telegram bridge for the [pi coding agent](https://github.com/badlogic/pi-mono) with first-class UX (one-step setup, rich text, native dialog buttons).

**Audience:** International. **All** code, comments, identifiers, docs, README, commit messages, and Telegram-facing strings are in **English**.

**Architecture:** Deep modules behind small interfaces at clean seams (per `codebase-design`). A thin pi-extension entrypoint composes pure domain logic with three swappable adapters (Telegram transport, pi-agent, config store). No daemon, no service installer, no handoff — the bridge lives only while the pi session is alive. Source in TypeScript, built to ESM that runs on **Node 20+ and Bun**.

**Tech Stack:** TypeScript, Bun (dev/test/build), Node 20+ runtime target, `marked` (markdown→HTML), peer deps `@earendil-works/{pi-ai,pi-agent-core,pi-coding-agent}@^0.80` + `@sinclair/typebox`. Tests: `bun test`. Commit hygiene: husky + commitlint (Conventional Commits).

---

## Decisions locked (this revision)

- **Dropped:** multi-user allowlist (back to single-user first-DM pairing), external prompt inbox.
- **Deferred to v0.2:** voice transcription (needs paid Whisper API + key — conflicts with "easy setup"), per-topic sessions.
- **Kept as headline UX:** one-step setup, rich-text replies, streaming previews, native inline-keyboard dialogs (select/confirm/text).
- **Session-local only.** No launchd/systemd, no `/handoff`–`/handback`.
- **Breaking config change** from old `telegram.json` is acceptable; ship a non-destructive one-time migration.

## Method (how we honor the skills)

- **domain-modeling first.** Before any module, write `CONTEXT.md` (glossary) + ADRs for hard-to-reverse calls. Module names and test names use this ubiquitous language.
- **codebase-design.** Apply the **deletion test** to every proposed module; reject pass-throughs. Prefer **deep** modules (small interface, lots behind it). Introduce a seam only when something actually varies across it — each seam ships with a real adapter **and** an in-memory fake (the test surface). The old 1763-line monolith is the canonical shallow-sprawl we are deepening.
- **tdd — vertical slices.** One test → minimal impl → next test. NEVER "write all tests, then all code." Tests exercise public interfaces only, so they survive refactors. Never refactor while red.
- **Conventional Commits**, enforced by commitlint. Frequent, small commits — one per task.

## Ubiquitous Language (seed for CONTEXT.md — refine in Task 0.3)

| Term | Meaning (glossary, no implementation) |
|---|---|
| **Bridge** | The live link between one Telegram bot and one pi session. Exists only while connected. |
| **Pairing** | The first-DM handshake that binds the bot to a single allowed Telegram user. |
| **Session** | The underlying pi coding-agent session the Bridge drives. |
| **Turn** | One user prompt → assistant reply cycle, including queued follow-ups and attachments. |
| **Prompt** | A Telegram message mapped into pi input (text, images, file paths), prefixed `[telegram]`. |
| **Preview** | The streaming, partial assistant text shown while pi is still generating. |
| **Dialog** | A native Telegram inline-keyboard interaction (select/confirm/text) a pi command can request mid-turn. |
| **Attachment** | An outbound file pi sends back to Telegram via the `telegram_attach` tool. |
| **Config** | User-edited settings (token + UX flags). Distinct from **State**. |
| **State** | Machine-managed runtime cursor (`lastUpdateId`, cached bot identity). Never hand-edited. |
